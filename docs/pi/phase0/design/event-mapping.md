# Pi RPC To T3 Event Mapping

Draft mapping for Phase 5 implementation. Refined from `pi-plan.md`, upstream RPC docs, and local `pi 0.72.1` probe output.

| Pi RPC event/response                                         | T3 runtime event                                                                           | Mapping policy                                                                                                                                                  |
| ------------------------------------------------------------- | ------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `response` with `command: "prompt"`, `success: true`          | internal accepted-turn marker                                                              | Command response means accepted/queued/handled. It is not the terminal result. Start watchdog after this point.                                                 |
| `response` with `success: false`                              | `turn.completed` with `state: "failed"` or pre-accept validation failure                   | If failure occurs before prompt acceptance, fail send operation or emit terminal failure if a T3 turn id was already allocated.                                 |
| `agent_start`                                                 | internal active-agent marker                                                               | Do not emit duplicate `turn.started` if `turn_start` follows.                                                                                                   |
| `turn_start`                                                  | `turn.started`                                                                             | Emit once per T3 user prompt. Ignore or fold additional native Pi turn starts within the same accepted prompt unless Phase 5 proves a better mapping.           |
| `message_start` role `user`                                   | usually none                                                                               | T3 already has the user message. Raw-log only unless needed for reconciliation.                                                                                 |
| `message_start` role `assistant`                              | internal assistant message open                                                            | Do not emit a blank assistant item until there is text, reasoning, or tool content.                                                                             |
| `message_update.assistantMessageEvent.type: "text_start"`     | internal content block open                                                                | Prepare assistant text stream for `contentIndex`.                                                                                                               |
| `message_update.assistantMessageEvent.type: "text_delta"`     | `content.delta` with `streamKind: "assistant_text"`                                        | Emit `delta`; preserve `contentIndex` when present.                                                                                                             |
| `message_update.assistantMessageEvent.type: "text_end"`       | internal content block close                                                               | No separate T3 event unless needed for snapshot accumulation.                                                                                                   |
| `message_update.assistantMessageEvent.type: "thinking_start"` | internal reasoning block open                                                              | Prepare reasoning stream.                                                                                                                                       |
| `message_update.assistantMessageEvent.type: "thinking_delta"` | `content.delta` with `streamKind: "reasoning_text"`                                        | Emit only textual thinking delta; redact/avoid persisting encrypted thinking signatures.                                                                        |
| `message_update.assistantMessageEvent.type: "thinking_end"`   | internal reasoning block close                                                             | Do not persist encrypted reasoning signatures in fixtures/logs unless privacy policy allows.                                                                    |
| `message_update.assistantMessageEvent.type: "toolcall_start"` | `item.started`                                                                             | Map to best canonical tool item type if tool name is known; otherwise `dynamic_tool_call`.                                                                      |
| `message_update.assistantMessageEvent.type: "toolcall_delta"` | `item.updated` or `content.delta`                                                          | Prefer `item.updated` for structured tool-call arguments; avoid duplicate emission if `tool_execution_*` events also arrive for the same tool.                  |
| `message_update.assistantMessageEvent.type: "toolcall_end"`   | internal or `item.completed`                                                               | Coordinate with `tool_execution_end`; do not double-complete.                                                                                                   |
| `tool_execution_start`                                        | `item.started`                                                                             | Use canonical item type (`command_execution`, `file_change`, `mcp_tool_call`, `dynamic_tool_call`, etc.) based on tool metadata.                                |
| `tool_execution_update`                                       | `item.updated`, `tool.progress`, or `content.delta`                                        | Stream tool output/progress. Command stdout/stderr can map to `content.delta` with `command_output` if the UI expects textual output.                           |
| `tool_execution_end`                                          | `item.completed`                                                                           | Include status, summary, and result metadata.                                                                                                                   |
| `message_end` role `assistant`                                | internal assistant message close                                                           | Still do not complete the T3 turn here; wait for `agent_end` or reconciled native idle.                                                                         |
| `turn_end`                                                    | internal native turn ended, fallback terminal marker                                       | Use as fallback if `agent_end` is absent and `get_state.isStreaming` is false. Include stop reason/error if present.                                            |
| `agent_end`                                                   | `turn.completed`                                                                           | Preferred terminal success/failure event. If assistant `stopReason` is `aborted`, emit `turn.aborted` or completed-interrupted according to adapter convention. |
| `queue_update`                                                | `runtime.warning` or internal queue status                                                 | Useful if T3 supports queued follow-ups later. Not required for conservative v1 mid-turn reject policy.                                                         |
| `compaction_start`                                            | `runtime.warning` or `thread.state.changed`                                                | Do not complete the turn.                                                                                                                                       |
| `compaction_end`                                              | `runtime.warning` / internal                                                               | If `willRetry` or retry metadata indicates another attempt, keep turn active.                                                                                   |
| `auto_retry_start`                                            | `runtime.warning`                                                                          | Keep T3 turn active.                                                                                                                                            |
| `auto_retry_end`                                              | `runtime.warning` or `turn.completed` failed on final failure                              | Use final failure if Pi reports retry exhausted and no later completion arrives.                                                                                |
| `extension_ui_request` dialog                                 | `user-input.requested`                                                                     | Must store pending request and later write `extension_ui_response` to stdin from `respondToUserInput`.                                                          |
| `extension_ui_request` fire-and-forget                        | `runtime.warning` or raw log                                                               | Do not block; no stdin response expected.                                                                                                                       |
| process `exit` while idle                                     | `session.exited`                                                                           | Include recoverable flag and diagnostics.                                                                                                                       |
| process `exit` mid-turn                                       | `session.exited` plus `turn.completed` failed                                              | Reject pending commands and emit terminal failure so UI cannot stay `working...`.                                                                               |
| `abort` response success                                      | `turn.aborted` after stream confirms abort, or immediate abort marker if no stream arrives | Local probe showed abort response can arrive after `turn_end`/`agent_end`; adapter should tolerate either order.                                                |

## Raw Logging

Use `raw.source` subcategories:

- `pi.rpc.response`
- `pi.rpc.event`
- `pi.rpc.extension_ui`
- `pi.rpc.stderr`
- `pi.rpc.process`

Persisted raw logs must redact secrets, local paths, response ids, encrypted reasoning signatures, and extension payloads that may contain private user input.
